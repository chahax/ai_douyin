# AI Douyin 项目演进路线图

更新时间：2026-03-29

## 1. 这份文档现在怎么理解

这份文档原本偏向“可部署服务改造路线图”，但结合当前项目目标，已经调整为“双路线说明”：

- 主路线：功能完成优先
- 次路线：服务化后置

也就是说，当前更推荐先把项目做成一个真正可用的抖音内容生产工具，再决定是否继续把它包装成 API 服务。

## 2. 当前判断

当前项目已经完成：

- 阶段 0：基础加固
- 阶段 1：应用服务层抽取
- 阶段 2：Ollama 接入

所以现在不再建议直接进入“阶段 3 API 服务层”，而是先补业务功能闭环。

## 3. 推荐执行顺序

推荐按下面这个顺序推进：

1. 完整视频生成
2. 抖音自动发布
3. 评论获取与平台回收
4. 产物记录与失败重试
5. 如果仍有必要，再进入 API / 队列 / Worker / 部署

## 4. 路线 A：功能完成优先

### A1. 完整视频生成

目标：

- 把当前“音频成品”升级为“短视频成品”

建议交付物：

- `video_composer.py` 真正落地
- 支持背景图 / 背景视频
- 支持字幕叠加
- 支持封面或片头
- 支持固定尺寸输出，如 9:16

最低完成标准：

- 输入一段文案后，最终能产出可直接发布的竖版视频文件

### A2. 抖音自动发布

目标：

- 让项目从“生成内容”走到“平台发布”

建议交付物：

- 抖音发布适配器
- 发布参数模型
- 标题、描述、话题、封面等字段映射
- 发布结果记录
- 发布失败重试或失败原因记录

最低完成标准：

- 能从本地生成产物触发一次稳定发布，并获得平台返回结果

### A3. 评论获取与回收

目标：

- 让项目从“只发不收”走到“能感知平台反馈”

建议交付物：

- 评论抓取入口
- 评论拉取数据结构
- 评论持久化
- 视频与评论关联
- 后续自动回复接口预留

最低完成标准：

- 能根据视频或平台资源 ID 拉回评论，并落到本地记录

### A4. 产物治理与运行记录

目标：

- 让生成、发布、评论这三段都能追踪

建议交付物：

- 任务日志
- 产物记录
- 发布记录
- 评论记录
- 失败原因记录

最低完成标准：

- 每个视频产物都能反查：怎么生成、是否发布、评论是否已同步

## 5. 路线 B：服务化后置

这条路线不是取消，而是后置。

更适合在下面任一场景出现时再推进：

- 你想做网页后台
- 你想做多端调用
- 你想做定时任务中心
- 你想让别人或其他程序也调用这套能力

### B1. API 服务层

目标：

- 把当前能力包装成 HTTP 接口

建议交付物：

- FastAPI 应用
- `GET /health`
- `POST /jobs/generate`
- `POST /jobs/generate-from-text`
- `POST /knowledge/import`

### B2. 异步任务执行

目标：

- 解耦长耗时任务

建议交付物：

- 队列
- Worker
- 任务状态管理

### B3. 元数据数据库

目标：

- 让服务化形态下的任务和产物可查询

建议交付物：

- jobs 表
- artifacts 表
- publish_records 表
- comments 表

### B4. 打包与部署

目标：

- 形成稳定可复现的部署方式

建议交付物：

- Dockerfile
- compose
- 环境配置说明

## 6. 为什么这样排

当前更推荐“功能优先”的原因是：

- 你主要是自己使用，不是先服务其他系统
- 你当前最缺的是完整业务闭环，不是调用方式
- API、队列、Worker 本身不会直接提升视频产物质量和平台运营价值
- 视频生成、自动发布、评论回收会直接提升项目完成度

## 7. 现在最合适的下一步

如果按当前路线推进，下一步最合适的是：

1. 先实现完整视频生成
2. 视频链路稳定后，再做抖音自动发布
3. 发布链路稳定后，再做评论获取

## 8. 一句话结论

这份路线图现在的核心意思是：

先把项目做成“能生成完整视频、能自动发布、能获取评论”的实用工具，再决定是否把它进一步服务化。
